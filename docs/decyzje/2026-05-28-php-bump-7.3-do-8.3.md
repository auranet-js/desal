# PHP bump 7.3.33 → 8.3.31 + CI 3.1.5 → 3.1.13

**Data:** 2026-05-28
**Wykonawca:** Auranet (Jan Schenk + Claude Code)
**Budżet:** ~3h z 6h budżetu pakietowego PHP bump (zadanie 2 z 4 w pakiecie 14h, akcept klienta 28.05)

## Decyzja

Migracja środowiska PHP z **7.3.33** (EOL 2021-12-06) do **8.3.31** (LTS do 2027-11) na produkcji desal.pl. W pakiecie z upgradem CodeIgniter z **3.1.5** do **3.1.13** (pierwsza wersja CI 3.x oficjalnie wspierająca PHP 8 to 3.1.12).

## Kontekst

PHP 7.3 nie otrzymuje patchy bezpieczeństwa od 4 lat. Każda znana CVE w PHP 7.3 z lat 2022-2026 pozostawała niezałatana. Bump był pierwszym z 4 zadań w zaakceptowanym 28.05 pakiecie prac (Otomoto fix, PHP bump, SEO basics, panel Otomoto enhancement).

Audyt techniczny 27.05 (`TECH_AUDIT_2026-05-27.md`) wskazał na potrzebę migracji, plus wykrył dodatkowo `php_flag display_errors on` w `.htaccess` — security finding na produkcji rozwiązany w pakiecie tego samego deploymentu.

## Kolejność i decyzje wykonawcze

**Wariant wybrany: (A) PHP bump przed Otomoto fix** — pierwotnie Claude rekomendował wariant (B) (Otomoto first, szybki client-visible win), ale klient zdecydował (A) z argumentem porządku architektonicznego.

**Tooling rozszerzony — MCP v1.2.0:** Pierwotnie planowano klikanie w panelu phpMyAdmin (DB dump) i WinSCP (upload CI 3.1.13) przez Janka. W trakcie sesji uznano że to porażka projektowania ("MCP jako source-of-truth"). MCP rozbudowany z v1.1.0 do v1.2.0 o 5 nowych tooli: `db_export`, `archive_deploy`, `htaccess_write`, `exec_safe`, `file_diff`. Cały PHP bump przeprowadzony bez ręcznej akcji klienta. Inwestycja MCP rozbudowy (~45 min) zwrócona po pierwszym użyciu, plus reusable dla pozostałych DuoCMS w portfolio Auranet (LAGUZ, Victorini, Agria, ASEO, pozostałe).

## Wykonane zmiany

### 1. CodeIgniter 3.1.5 → 3.1.13

- **Plik wymieniany:** wszystkie pliki pod `system/` (framework). `application/` (kod klienta) nietknięty.
- **Pre-deploy walidacja:** SHA256 17 plików `system/core/*` na prod = SHA256 github CI 3.1.5 → potwierdzenie zerowych customizacji frameworka.
- **Deploy:** `archive_deploy` tool — pobranie tarballa z auratest.pl, automatyczny backup starego system/ jako `system.bak-20260528-135512.tar.gz` w site_root, extract z `--strip-components=1` i `--no-same-owner --no-same-permissions --no-overwrite-dir`.
- **Post-deploy walidacja:** SHA256 nowego `system/core/CodeIgniter.php` = SHA256 github CI 3.1.13. `CI_VERSION = '3.1.13'` w pliku.

### 2. PHP 7.3.33 → 8.3.31

- **Metoda:** `.htaccess` directive `AddHandler application/x-httpd-alt-php83___lsphp .php` na CloudLinux LVE + LiteSpeed. **Nie wymagało dotykania panelu hostingu** — pełen switch przez MCP.
- **Pre-bump backup:** `.htaccess.bak-20260528-132556` (przed pierwszym backupem strategicznym) + `.htaccess.bak-20260528-135639` (przed faktycznym switchem).
- **Walidacja:** PHP 8.3.31 SAPI litespeed, `include_path: /opt/alt/php83/...` confirmed.

### 3. Security cleanup `.htaccess`

W jednym kroku z PHP switchem usunięte:
- `php_flag display_errors on` — security finding z audytu (PHP errors wyświetlane na produkcji)
- `php_value suhosin.*` (3 wpisy) — legacy z PHP 5.x ery, suhosin nie istnieje w PHP 7+/8

Zostawione:
- `php_value max_input_vars 3000`
- CodeIgniter pretty URL rewriterules

### 4. MCP desal-duocms v1.1.0 → v1.2.0

Dodane tooly (z `MCP_ROADMAP.md` grupa C):
- `db_export` — pełen mysqldump → gzip → zapis w katalogu MCP, pobranie przez HTTPS
- `archive_deploy` — pobranie tarballa z whitelist URL (auratest.pl, github), backup starego target, extract z safe flags
- `htaccess_write` — kontrolowany write `.htaccess` z auto-backup (omija anti-suicide blokadę write_file)
- `exec_safe` — whitelisted shell exec (mysqldump, tar, gzip, which, head, tail, du, wc, php, find, grep, stat, file, date, ...). Niewerybalna komenda `escapeshellarg` na argumentach.
- `file_diff` — SHA256 readout albo porównanie z expected_sha256/expected_content (integrity check bez wczytywania pełnego pliku)

Nowe whitelisty:
- `MCP_DEPLOY_WHITELIST`: auratest.pl, raw.githubusercontent.com, github.com, codeload.github.com
- `MCP_EXEC_WHITELIST`: 26 komend read/archive/diagnostyka

## Otwarte sprawy (poza scope tej decyzji)

1. **opcache wyłączony** — extension nie załadowane w altphp83 na dmkhosting. Włączenie wymaga supportu hostingu (lub może być w panelu dmkhosting jako PHP Selector option, do sprawdzenia). Performance gain typowo 30-50% przy 27k produktów — wart kontaktu z dmkhosting.
2. **Status MCP** pokazuje `codeigniter_version: null` mimo działającego frameworka. Regex w `tool_status` matchuje stary CI `define()`-style stałe; CI 3.1.13 używa `const CI_VERSION = '3.1.13';`. Drobny bug do fixu w MCP v1.2.1.
3. **Stary `system.bak-20260528-135512.tar.gz`** zostaje na serwerze klienta — sugeruję pozostawić 30 dni jako safety net, potem usunąć (przez MCP `rm`).
4. **WordPress remnant** `/wp-content/` w site_root (z 2014-2022, nieaktywny) — osobny temat sprzątania, nie dotyczy PHP bumpu.
5. **Charset niespójny** (latin2 mieszany z utf8mb3 w niektórych tabelach) — osobne zadanie z audytu, nie dotyczy PHP bumpu.

## Lessons learned (do `_duocms-playbook/`)

1. **Tarball z entry `./` może zmienić uprawnienia katalogu docelowego.** Doświadczone w trakcie sesji — site_root chmod zmieniło się z 755 na 700 po pierwszym uruchomieniu `archive_deploy` bez `--no-same-permissions`. Apache zwrócił HTTP 404 na cały MCP. Fix: zawsze `tar xzf ... --no-same-owner --no-same-permissions --no-overwrite-dir`.
2. **CloudLinux altphp PHP switch przez `.htaccess`** — `AddHandler application/x-httpd-alt-phpXX___lsphp .php`. Działa bez restartu, natychmiastowy efekt. Wzorzec reusable dla wszystkich DuoCMS na CloudLinux + LiteSpeed.
3. **CI 3.1.5 → 3.1.13** — `application/` zostaje nietknięty (kod klienta), tylko `system/` (framework) wymieniany. SHA256 walidacja przed deploymentem potwierdziła zerowe customizacje frameworka — to pattern reusable dla pozostałych DuoCMS gdzie CI był podobnie zarządzany.
4. **MCP grupa C tooli** to inwestycja jednorazowa per DuoCMS klient ALE krócącą się drastycznie z każdym kolejnym projektem. Pakiet 5 tooli zbudowany w ~45 min, zwrócił się od razu w tej sesji, plus reusable dla LAGUZ/Victorini/Agria/ASEO/pozostałych.
